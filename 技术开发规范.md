## 买家说 — 技术开发规范

> 本文档定义项目从第一行代码开始就必须遵循的工程规范。
> 参考来源：Next.js 官方最佳实践、Google Web Fundamentals、阿里前端开发手册、小红书/字节跳动工程规范。
> 所有规范均为强制项，Code Review 时逐条检查。

**规则**：
- 禁止在 `components/` 下放页面级组件
- 禁止在 `app/` 下放可复用组件
- 每个 `app/[feature]/` 最多包含 `page.tsx`（服务端）+ `client.tsx`（客户端）+ `loading.tsx` + `error.tsx`
- `lib/` 只放无业务逻辑的工具函数和第三方封装
- `services/` 只放与后端交互的请求函数
- 文件不超过 **300 行**，超过必须拆分

### 二、命名规范

#### 2.1 文件命名

| 类型 | 规则 | 示例 |
|------|------|------|
| 页面 | `page.tsx` / `client.tsx` | `app/post/[id]/page.tsx` |
| 组件 | PascalCase | `PostCard.tsx` |
| Hook | camelCase + use 前缀 | `usePosts.ts` |
| 工具函数 | camelCase | `formatTime.ts` |
| 类型定义 | camelCase | `post.ts` |
| 测试文件 | `*.test.ts(x)` | `PostCard.test.tsx` |
| 常量文件 | camelCase | `constants.ts` |
| 样式文件 | 禁止独立 CSS 文件 | 全部使用 Tailwind 类 |

#### 2.2 代码命名

| 类型 | 规则 | 示例 |
|------|------|------|
| 组件 | PascalCase | `PostCard`, `ImageUploader` |
| 函数/变量 | camelCase | `fetchPosts`, `activeTag` |
| 常量 | UPPER_SNAKE_CASE | `MAX_IMAGES`, `PAGE_SIZE` |
| 类型/接口 | PascalCase | `Post`, `UserProfile` |
| 枚举 | PascalCase + PascalCase 成员 | `PostStatus.Active` |
| CSS 类 | kebab-case（Tailwind 自定义） | `tag-chip-active` |
| 事件处理 | handle + 动作 | `handleSubmit`, `handleLike` |
| 布尔变量 | is/has/can/should 前缀 | `isLoading`, `hasMore`, `canEdit` |
| Ref | 名词 + Ref 后缀 | `carouselRef`, `inputRef` |

#### 2.3 禁止命名

- 单字母变量（循环 `i` 除外）
- 拼音命名（用英文）
- 缩写歧义（`usr` → `user`，`mgr` → `manager`）
- `_` 前缀私有变量（TypeScript 用 `private` 关键字）
- `data1`, `data2`, `temp`, `info` 等无意义名称

### 三、TypeScript 规范

#### 3.1 严格模式（tsconfig.json）

```json
{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "noImplicitReturns": true,
    "noFallthroughCasesInSwitch": true,
    "forceConsistentCasingInFileNames": true
  }
}
```

#### 3.2 类型定义规则

| 场景 | 使用 | 理由 |
|------|------|------|
| 数据模型（Post, User, Comment） | `interface` | 可扩展、可合并 |
| 联合类型 / 字面量 | `type` | `type Status = 'active' \| 'hidden'` |
| 函数参数超过 3 个 | `interface` 封装 | `interface FetchPostsParams { ... }` |
| 组件 Props | `interface Props` | 统一命名，便于扩展 |
| API 响应 | `interface` + 泛型 | `interface ApiResponse<T> { code: number; data: T }` |
| 枚举 | `const` 对象 + type | 禁止使用 `enum` 关键字 |

#### 3.3 禁止项

```typescript
// ❌ 禁止 any
function fetchData(): any { ... }

// ✅ 使用 unknown + 类型守卫
function fetchData(): Promise<unknown> { ... }

// ❌ 禁止 @ts-ignore
// @ts-ignore
const x = someVar.y

// ✅ 使用可选链 + 空值合并
const x = someVar?.y ?? defaultValue

// ❌ 禁止非空断言滥用
const post = posts[0]!

// ✅ 使用条件判断
const post = posts[0]
if (!post) return null

// ❌ 禁止类型断言绕过
const data = response as Post

// ✅ 使用类型守卫
function isPost(obj: unknown): obj is Post { ... }
```

#### 3.4 泛型约定

```typescript
// API 统一响应
interface ApiResponse<T> {
  code: number
  message: string
  data: T
}

// 分页响应
interface PaginatedResponse<T> {
  list: T[]
  total: number
  hasMore: boolean
  page: number
}

// Hook 返回值
interface UseQueryResult<T> {
  data: T | null
  loading: boolean
  error: Error | null
  refetch: () => void
}
```

### 四、组件设计规范

#### 4.1 服务端 vs 客户端组件

| 场景 | 组件类型 | 文件 |
|------|----------|------|
| 数据获取 + SEO + 无交互 | Server Component | `page.tsx` |
| useState / useEffect / 事件处理 | Client Component | `client.tsx` |
| 纯展示 + 无状态 | Server Component | `page.tsx` |
| Framer Motion 动画 | Client Component | `client.tsx` |

**原则**：默认用 Server Component，只在需要交互时添加 `'use client'`。

#### 4.2 Props 设计

```typescript
// ✅ Props 用 interface，必须定义默认值或 required
interface PostCardProps {
  post: Post
  index?: number        // 可选，默认 0
  onLike?: (id: string) => void  // 回调可选

// ✅ 超过 5 个 props 必须分组
interface PublishFormProps {
  // 数据
  initialData?: DraftPost

  onCancel: () => void

// ❌ 禁止 children: any
// ✅ 使用 ReactNode 或明确类型
interface ModalProps {
  children: React.ReactNode
  title: string
  onClose: () => void
}
```


#### 4.4 组件拆分规则

- **单个 JSX 返回 > 80 行** → 拆子组件
- **同一段逻辑出现 2 次** → 抽 Hook
- **同一 UI 模式出现 3 次** → 抽原子组件到 `components/ui/`
- **组件 props > 7 个** → 拆分为多个子组件

### 五、状态管理规范

#### 5.1 状态选型矩阵

| 状态类型 | 存储位置 | 工具 | 示例 |
|----------|----------|------|------|
| UI 临时状态 | 组件内 | `useState` | 输入框文字、展开/折叠 |
| 表单状态 | 组件内 | `useState` / `useReducer` | 发布表单字段 |
| 服务端数据 | 服务端组件 / SWR | Next.js fetch / SWR | 帖子列表、用户信息 |
| 全局用户态 | Zustand | `useUserStore` | 登录状态、用户资料 |
| 全局 UI 态 | Zustand | `useUIStore` | Toast 队列、Modal 开关 |
| URL 参数 | URL | `useSearchParams` | 搜索关键词、筛选标签 |
| 持久化数据 | localStorage | 自定义 Hook | 搜索历史、主题偏好 |

#### 5.2 Zustand Store 规范

```typescript
// store/user.ts
import { create } from 'zustand'
import type { User } from '@/types'

// ✅ 1. 导出 interface
interface UserState {
  user: User | null
  isLoggedIn: boolean
  // ✅ 2. action 用动词
  setUser: (user: User | null) => void
  logout: () => void
  updateUser: (patch: Partial<User>) => void
}

// ✅ 3. 单一 store 文件，不拆分 slice
export const useUserStore = create<UserState>((set) => ({
  user: null,
  isLoggedIn: false,
  setUser: (user) => set({ user, isLoggedIn: !!user }),
  logout: () => set({ user: null, isLoggedIn: false }),
  updateUser: (patch) => set((state) => ({
    user: state.user ? { ...state.user, ...patch } : null,
  })),
}))
```

**规则**：
- 全局 store 不超过 3 个（user / ui / 业务）
- store 内禁止异步操作，异步放 `services/` 层
- 组件内用 `useUserStore((s) => s.user)` 精确订阅，禁止 `useUserStore()` 全量订阅

#### 6.2 错误处理规范（强制）

错误处理是工程质量的底线。本节规定**唯一**的错误抛出形式、错误码来源、用户提示形式与日志格式，所有 `services/` 与组件层必须严格遵守。

##### 6.2.1 错误码统一配置

所有业务错误码集中维护在 `src/lib/error-codes.ts`，禁止在业务代码中硬编码字符串错误信息或散落定义。

```typescript
// src/lib/error-codes.ts
export const ErrorCode = {
  // ─── 通用 (1xxx) ───
  UNKNOWN:           { code: 1000, message: '未知错误，请稍后重试' },
  NETWORK:           { code: 1001, message: '网络异常，请检查网络后重试' },
  TIMEOUT:           { code: 1002, message: '请求超时，请稍后重试' },

  // ─── 认证 (2xxx) ───
  UNAUTHORIZED:      { code: 2001, message: '请先登录' },
  TOKEN_EXPIRED:     { code: 2002, message: '登录已过期，请重新登录' },
  FORBIDDEN:         { code: 2003, message: '没有权限执行该操作' },

  // ─── 校验 (3xxx) ───
  PARAM_INVALID:     { code: 3001, message: '参数不合法' },
  TITLE_TOO_SHORT:   { code: 3002, message: '标题需 2-50 字' },
  CONTENT_TOO_SHORT: { code: 3003, message: '正文至少 10 字' },
  CONTENT_TOO_LONG:  { code: 3004, message: '正文最多 2000 字' },

  // ─── 资源 (4xxx) ───
  NOT_FOUND:         { code: 4001, message: '资源不存在' },
  POST_NOT_FOUND:    { code: 4002, message: '帖子不存在或已删除' },
  ALREADY_EXISTS:    { code: 4003, message: '资源已存在' },

  // ─── 业务 (5xxx) ───
  POST_CREATE_FAIL:  { code: 5001, message: '发布失败，请稍后重试' },
  COMMENT_FAIL:      { code: 5002, message: '评论失败，请稍后重试' },
  LIKE_FAIL:         { code: 5003, message: '操作失败，请稍后重试' },
  UPLOAD_FAIL:       { code: 5004, message: '图片上传失败，请重试' },
} as const

export type ErrorCodeKey = keyof typeof ErrorCode
```

**规则**：
- 错误码按业务域分段（通用 1xxx / 认证 2xxx / 校验 3xxx / 资源 4xxx / 业务 5xxx）
- 新增错误码必须更新本表，禁止散落在 `services/` 中拼字符串
- `message` 即默认用户文案，UI 层无需再翻译；如需差异化文案，调用方传入 override

##### 6.2.2 统一异常类 `ApiError`

所有由 `services/` 主动抛出的错误，必须为 `ApiError` 实例，禁止抛 `new Error()` 或 `string`。

```typescript
// src/lib/api-error.ts
import { ErrorCode, type ErrorCodeKey } from './error-codes'

export class ApiError extends Error {
  /** 业务错误码（来自 ErrorCode 表） */
  public readonly code: number
  /** 给用户看的提示文案 */
  public readonly userMessage: string
  /** 原始错误（Supabase / 网络层抛出的） */
  public readonly cause?: unknown

  constructor(key: ErrorCodeKey, options?: { userMessage?: string; cause?: unknown }) {
    const { code, message } = ErrorCode[key]
    super(`[${code}] ${message}`)
    this.name = 'ApiError'
    this.code = code
    this.userMessage = options?.userMessage ?? message
    this.cause = options?.cause
  }

  /** 类型守卫 */
  static is(err: unknown): err is ApiError {
    return err instanceof ApiError
  }
}
```

##### 6.2.3 services 层抛错形式（强制）

```typescript
// services/post.ts
import { supabase } from '@/lib/supabase'
import { ApiError } from '@/lib/api-error'

export async function createPost(params: CreatePostParams): Promise<Post> {
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) throw new ApiError('UNAUTHORIZED')

  // 前置校验，与数据库 CHECK 约束镜像
  if (params.title.trim().length < 2) throw new ApiError('TITLE_TOO_SHORT')
  if (params.content.trim().length < 10) throw new ApiError('CONTENT_TOO_SHORT')

  const { data, error } = await supabase
    .from('posts')
    .insert({ user_id: user.id, ...params })
    .select()
    .single()

  if (error) {
    throw new ApiError('POST_CREATE_FAIL', { cause: error })
  }
  return data as Post
}
```

##### 6.2.4 调用方处理形式（强制）

业务代码（页面/组件/Hook）统一用以下三步处理：

```typescript
import { ApiError } from '@/lib/api-error'
import { useUIStore } from '@/store/ui'

const handleSubmit = async () => {
  try {
    setSubmitting(true)
    const post = await createPost(form)
    useUIStore.getState().addToast('success', '发布成功')
    router.push(`/post/${post.id}`)
  } catch (err) {
    // 1. ApiError → 直接展示 userMessage
    if (ApiError.is(err)) {
      useUIStore.getState().addToast('error', err.userMessage)
      return
    }
    // 2. 其它未知错误 → 展示通用兜底文案 + 上报
    useUIStore.getState().addToast('error', '操作失败，请稍后重试')
    console.error('[unhandled]', err)
  } finally {
    setSubmitting(false)
  }
}
```

##### 6.2.5 用户提示分级

| 错误类型 | 提示方式 | 示例 |
|---------|---------|------|
| 表单校验（用户实时输入） | 内联提示（红字 + 按钮上方 hint） | 「使用体验还需 X 字」 |
| 业务可恢复错误（点赞/评论失败） | Toast（自动消失） | 「评论失败，请稍后重试」 |
| 认证类（未登录/过期） | 弹起登录浮层 | 跳转 `/login` 或 LoginSheet |
| 阻断性错误（详情不存在） | 整页错误态 + 返回按钮 | 「帖子不存在或已删除」 |
| 全局兜底（未捕获异常） | `app/error.tsx` 错误边界 | 「页面出错了，刷新试试」 |

**禁止**：
- 把后端原始 `error.message` 直接 toString 后展示给用户（如 PostgREST 裸约束名）
- 同一错误重复弹多个 Toast（队列里同 message 去重）
- 在 `services/` 层弹 Toast（弹 Toast 是 UI 关注点，必须由调用方处理）

##### 6.2.6 日志规范

```typescript
// 仅在 catch 未知分支或上报点使用
console.error('[scope]', err)              // ✅ 带 scope 前缀
console.error('[publish] createPost', err) // ✅ 便于线上日志检索

console.log('debug:', data)                // ❌ 生产代码禁止 console.log
```

**规则**：
- 生产构建禁止 `console.log`（ESLint `no-console: ['error', { allow: ['warn', 'error'] }]`）
- `console.error` 必须带 `[scope]` 前缀
- 用户敏感信息（手机号、邮箱、token）禁止打日志
- 后续接入 Sentry / 阿里 ARMS 后，未知分支 `console.error` 改为上报

##### 6.2.7 禁止项

```typescript
// ❌ 抛字符串
throw '发布失败'

// ❌ 抛裸 Error
throw new Error('发布失败')

// ❌ 在 services 层弹 Toast
export async function createPost(...) {
  if (error) {
    useUIStore.getState().addToast('error', '失败')  // 错！UI 关注点泄漏到 service 层
    throw error
  }
}

// ❌ 吞错
try { await createPost(...) } catch {}

// ❌ 把 Supabase 原始 error 直接给用户看
catch (err) {
  alert(err.message) // 可能是 'new row for relation "posts" violates check constraint "posts_content_check"'
}

// ❌ 错误码硬编码
throw new Error('E_3002: 标题太短')  // 应该用 new ApiError('TITLE_TOO_SHORT')
```

#### 9.1 XSS 防护

| 场景 | 防护方式 |
|------|----------|
| 用户输入展示 | React 默认转义 JSX，禁止 `dangerouslySetInnerHTML` |

| 规则 | 说明 |
|------|------|
| Token 存储 | Supabase SDK 自动管理，禁止手动存 cookie |

| 文件上传 | 服务端校验 MIME type + 文件大小（≤ 10MB） |
| SQL 注入 | Supabase SDK 自动参数化查询，禁止拼接 SQL |

#### 9.3 输入校验

```typescript
// lib/validators.ts
export const validators = {
  title: (v: string) => v.trim().length >= 2 && v.trim().length <= 50,
  content: (v: string) => v.trim().length >= 10 && v.trim().length <= 2000,
  nickname: (v: string) => /^[\u4e00-\u9fa5a-zA-Z0-9_]{2,20}$/.test(v),
  price: (v: string) => /^¥?\d+(\.\d{1,2})?$/.test(v) || v === '',
}
```

**规则**：前端校验 = 用户体验，后端校验 = 安全底线，两者缺一不可。

### 十一、数据库规范

#### 11.1 字段类型选型

| 场景 | 推荐类型 | 反例 |
|------|----------|------|
| 状态/标志/枚举（如 post.status、user.role） | `SMALLINT` / `INT` + 代码常量 | ❌ `TEXT` + `CHECK (status IN ('a','b','c'))` |
| 计数器（如 like_count、post_count） | `INT` + 触发器维护 | ❌ 每次查询聚合 |
| 金额 | `INT`（以分为单位）或 `NUMERIC(10,2)` | ❌ `FLOAT` / `DOUBLE PRECISION`（精度丢失） |
| 时间戳 | `TIMESTAMPTZ`（带时区） | ❌ `TIMESTAMP`（无时区，跨区会乱） |
| ID | `UUID` + `gen_random_uuid()` | ❌ `BIGSERIAL`（暴露增长趋势） |
| 数组（如 tags） | `TEXT[]` | ❌ 关联表 + JOIN（除非需要独立管理） |

**核心原则**：状态等**简单含义**的字段尽量使用 `int` 类型存储，业务语义由代码常量维护，数据库不再用 `CHECK + 字符串枚举` 绑定具体值。好处：

- **迁移成本低**：状态枚举调整（如新增 `pending`）只改代码常量，不用 `ALTER TABLE` 改约束。
- **查询性能**：int 比较比字符串快，索引体积更小。
- **避免 typo 踩坑**：`WHERE status = 'actie'`（拼错）能编译过，但 `WHERE status = POST_STATUS.ACTIVE` 类型守护。
- **跨语言友好**：int 值在 Go/Java/Python 后端都能直接复用。

#### 11.2 代码侧常量维护

所有 DB 状态枚举必须在 `src/lib/constants.ts` 集中声明 `as const` 对象 + 配套 value 类型 + 可选中文标签：

```ts
/** 帖子状态（posts.status SMALLINT） */
export const POST_STATUS = {
  ACTIVE: 0,
  HIDDEN: 1,
  DELETED: 2,
} as const
export type PostStatusValue = (typeof POST_STATUS)[keyof typeof POST_STATUS]
export const POST_STATUS_LABEL: Record<PostStatusValue, string> = {
  [POST_STATUS.ACTIVE]: '正常',
  [POST_STATUS.HIDDEN]: '已隐藏',
  [POST_STATUS.DELETED]: '已删除',
}
```

**强制**：

- service 层 / page 层禁止在 Supabase 查询里写状态字符串字面量（`'active'`/`'deleted'`），必须用常量（`POST_STATUS.ACTIVE`）。
- mock 数据同步使用常量（`status: POST_STATUS.ACTIVE`），保持类型一致。
- 新增状态值时，同步更新：常量对象 → 类型 → 标签 → 文档（本小节 + 技术方案文档 §六）。

#### 11.3 迁移脚本规范

- 文件命名：`supabase/migration-v{N}-{feature}.sql`，N 单调递增。
- 脚本顶部注释必须包含：执行方式（Dashboard SQL Editor）、字段对照表（旧值 → 新值）、设计依据。
- 状态字段类型变更走「加临时列 → 数据迁移 → 删旧列 → 重命名 → 重建索引/策略」五步法，保证回滚可行。
- `SECURITY DEFINER` 函数内涉及状态字段的，必须与迁移同步更新（避免旧函数写新 schema 报错）。

#### 11.4 已落地的状态枚举

| 表 / 字段 | 类型 | 常量位置 | 取值 |
|-----------|------|----------|------|
| `posts.status` | `SMALLINT` | `POST_STATUS` | 0=正常 / 1=隐藏 / 2=删除 |
| `profiles.role` | `SMALLINT` | `USER_ROLE` | 0=普通用户 / 1=管理员 |
| `profiles.status` | `SMALLINT` | `USER_STATUS` | 0=正常 / 1=封禁 / 2=注销 |
| `tags.status` | `SMALLINT` | `TAG_STATUS` | 0=启用 / 1=已归档 / 2=已删除 |

> 全部状态/角色字段已统一为 SMALLINT + 代码常量。语义只在 `src/lib/constants.ts` 维护，DB 不再绑定字符串枚举语义；新增状态字段一律遵循该原则，禁止退回到 `TEXT + CHECK IN (...)` 模式。
