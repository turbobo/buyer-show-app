## 买家说 — 技术开发规范

> 本文档定义项目从第一行代码开始就必须遵循的工程规范。
> 参考来源：Next.js 官方最佳实践、Google Web Fundamentals、阿里前端开发手册、小红书/字节跳动工程规范。
> 所有规范均为强制项，Code Review 时逐条检查。

### 一、目录结构规范

```
buyer-show/
├── edgeone.json                  # 部署配置（禁止手动修改，由 CI 校验）
├── next.config.mjs               # Next.js 配置

├── postcss.config.mjs
├── tsconfig.json                 # 严格模式
├── .env.local                    # 本地环境变量（.gitignore）
├── .env.example                  # 环境变量模板（提交到仓库）
├── .eslintrc.json                # ESLint 规则

├── public/                       # 静态资源（favicon、manifest、og-image）

│   │       └── client.tsx        # 客户端组件（交互逻辑）
│   ├── components/               # 可复用 UI 组件
│   │   ├── ui/                   # 原子组件（Button, Input, Modal, Toast）
│   │   ├── layout/               # 布局组件（TabBar, TopNav, Sidebar）
│   │   └── [feature]/            # 业务组件（PostCard, CommentList）
│   ├── hooks/                    # 自定义 Hooks
│   ├── lib/                      # 第三方库封装 + 工具函数
│   │   ├── supabase.ts           # Supabase 客户端（单例）

│   │   ├── comment.ts            # 评论 CRUD
│   │   ├── user.ts               # 用户相关
│   │   └── auth.ts               # 认证相关
│   ├── store/                    # Zustand 状态管理
│   │   ├── user.ts               # 用户状态
│   │   └── ui.ts                 # UI 状态（toast, modal, sheet）

│   │   ├── post.ts               # 帖子相关类型
│   │   └── api.ts                # API 请求/响应类型
│   └── __tests__/                # 测试文件
│       ├── components/
│       ├── hooks/
│       └── services/
```

**规则**：
- 禁止在 `components/` 下放页面级组件
- 禁止在 `app/` 下放可复用组件
- 每个 `app/[feature]/` 最多包含 `page.tsx`（服务端）+ `client.tsx`（客户端）+ `loading.tsx` + `error.tsx`
- `lib/` 只放无业务逻辑的工具函数和第三方封装
- `services/` 只放与后端交互的请求函数
- 文件不超过 **300 行**，超过必须拆分

### 二、命名规范

#### 2.1 文件命名

| 类型 | 规则 | 示例 |
|------|------|------|
| 页面 | `page.tsx` / `client.tsx` | `app/post/[id]/page.tsx` |
| 组件 | PascalCase | `PostCard.tsx` |
| Hook | camelCase + use 前缀 | `usePosts.ts` |
| 工具函数 | camelCase | `formatTime.ts` |
| 类型定义 | camelCase | `post.ts` |
| 测试文件 | `*.test.ts(x)` | `PostCard.test.tsx` |
| 常量文件 | camelCase | `constants.ts` |
| 样式文件 | 禁止独立 CSS 文件 | 全部使用 Tailwind 类 |

#### 2.2 代码命名

| 类型 | 规则 | 示例 |
|------|------|------|
| 组件 | PascalCase | `PostCard`, `ImageUploader` |
| 函数/变量 | camelCase | `fetchPosts`, `activeTag` |
| 常量 | UPPER_SNAKE_CASE | `MAX_IMAGES`, `PAGE_SIZE` |
| 类型/接口 | PascalCase | `Post`, `UserProfile` |
| 枚举 | PascalCase + PascalCase 成员 | `PostStatus.Active` |
| CSS 类 | kebab-case（Tailwind 自定义） | `tag-chip-active` |
| 事件处理 | handle + 动作 | `handleSubmit`, `handleLike` |
| 布尔变量 | is/has/can/should 前缀 | `isLoading`, `hasMore`, `canEdit` |
| Ref | 名词 + Ref 后缀 | `carouselRef`, `inputRef` |

#### 2.3 禁止命名

- 单字母变量（循环 `i` 除外）
- 拼音命名（用英文）
- 缩写歧义（`usr` → `user`，`mgr` → `manager`）
- `_` 前缀私有变量（TypeScript 用 `private` 关键字）
- `data1`, `data2`, `temp`, `info` 等无意义名称

### 三、TypeScript 规范

#### 3.1 严格模式（tsconfig.json）

```json
{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "noImplicitReturns": true,
    "noFallthroughCasesInSwitch": true,
    "forceConsistentCasingInFileNames": true
  }
}
```

#### 3.2 类型定义规则

| 场景 | 使用 | 理由 |
|------|------|------|
| 数据模型（Post, User, Comment） | `interface` | 可扩展、可合并 |
| 联合类型 / 字面量 | `type` | `type Status = 'active' \| 'hidden'` |
| 函数参数超过 3 个 | `interface` 封装 | `interface FetchPostsParams { ... }` |
| 组件 Props | `interface Props` | 统一命名，便于扩展 |
| API 响应 | `interface` + 泛型 | `interface ApiResponse<T> { code: number; data: T }` |
| 枚举 | `const` 对象 + type | 禁止使用 `enum` 关键字 |

#### 3.3 禁止项

```typescript
// ❌ 禁止 any
function fetchData(): any { ... }

// ✅ 使用 unknown + 类型守卫
function fetchData(): Promise<unknown> { ... }

// ❌ 禁止 @ts-ignore
// @ts-ignore
const x = someVar.y

// ✅ 使用可选链 + 空值合并
const x = someVar?.y ?? defaultValue

// ❌ 禁止非空断言滥用
const post = posts[0]!

// ✅ 使用条件判断
const post = posts[0]
if (!post) return null

// ❌ 禁止类型断言绕过
const data = response as Post

// ✅ 使用类型守卫
function isPost(obj: unknown): obj is Post { ... }
```

#### 3.4 泛型约定

```typescript
// API 统一响应
interface ApiResponse<T> {
  code: number
  message: string
  data: T
}

// 分页响应
interface PaginatedResponse<T> {
  list: T[]
  total: number
  hasMore: boolean
  page: number
}

// Hook 返回值
interface UseQueryResult<T> {
  data: T | null
  loading: boolean
  error: Error | null
  refetch: () => void
}
```

### 四、组件设计规范

#### 4.1 服务端 vs 客户端组件

| 场景 | 组件类型 | 文件 |
|------|----------|------|
| 数据获取 + SEO + 无交互 | Server Component | `page.tsx` |
| useState / useEffect / 事件处理 | Client Component | `client.tsx` |
| 纯展示 + 无状态 | Server Component | `page.tsx` |
| Framer Motion 动画 | Client Component | `client.tsx` |

**原则**：默认用 Server Component，只在需要交互时添加 `'use client'`。

#### 4.2 Props 设计

```typescript
// ✅ Props 用 interface，必须定义默认值或 required
interface PostCardProps {
  post: Post
  index?: number        // 可选，默认 0
  onLike?: (id: string) => void  // 回调可选
}

// ✅ 超过 5 个 props 必须分组
interface PublishFormProps {
  // 数据
  initialData?: DraftPost
  // 回调
  onSubmit: (data: CreatePostParams) => void
  onCancel: () => void
  // 状态
  submitting?: boolean
}

// ❌ 禁止 children: any
// ✅ 使用 ReactNode 或明确类型
interface ModalProps {
  children: React.ReactNode
  title: string
  onClose: () => void
}
```

#### 4.3 组件结构模板

```typescript
'use client'

import { useState } from 'react'
import type { Post } from '@/types'

// 1. 类型定义
interface Props {
  post: Post
  index?: number
}

// 2. 动画 variants（Framer Motion）
const fadeUp = {
  hidden: { opacity: 0, y: 20 },
  visible: { opacity: 1, y: 0 },
}

// 3. 组件主体
export default function PostCard({ post, index = 0 }: Props) {
  // hooks（useState, useEffect, useRef, useMemo, useCallback）
  const [liked, setLiked] = useState(false)

  // 事件处理函数
  const handleLike = () => { ... }

  // 派生数据
  const displayCount = post.like_count > 999 ? '999+' : post.like_count

  // 渲染
  return (
    <div>...</div>
  )
}
```

#### 4.4 组件拆分规则

- **单个 JSX 返回 > 80 行** → 拆子组件
- **同一段逻辑出现 2 次** → 抽 Hook
- **同一 UI 模式出现 3 次** → 抽原子组件到 `components/ui/`
- **组件 props > 7 个** → 拆分为多个子组件

### 五、状态管理规范

#### 5.1 状态选型矩阵

| 状态类型 | 存储位置 | 工具 | 示例 |
|----------|----------|------|------|
| UI 临时状态 | 组件内 | `useState` | 输入框文字、展开/折叠 |
| 表单状态 | 组件内 | `useState` / `useReducer` | 发布表单字段 |
| 服务端数据 | 服务端组件 / SWR | Next.js fetch / SWR | 帖子列表、用户信息 |
| 全局用户态 | Zustand | `useUserStore` | 登录状态、用户资料 |
| 全局 UI 态 | Zustand | `useUIStore` | Toast 队列、Modal 开关 |
| URL 参数 | URL | `useSearchParams` | 搜索关键词、筛选标签 |
| 持久化数据 | localStorage | 自定义 Hook | 搜索历史、主题偏好 |

#### 5.2 Zustand Store 规范

```typescript
// store/user.ts
import { create } from 'zustand'
import type { User } from '@/types'

// ✅ 1. 导出 interface
interface UserState {
  user: User | null
  isLoggedIn: boolean
  // ✅ 2. action 用动词
  setUser: (user: User | null) => void
  logout: () => void
  updateUser: (patch: Partial<User>) => void
}

// ✅ 3. 单一 store 文件，不拆分 slice
export const useUserStore = create<UserState>((set) => ({
  user: null,
  isLoggedIn: false,
  setUser: (user) => set({ user, isLoggedIn: !!user }),
  logout: () => set({ user: null, isLoggedIn: false }),
  updateUser: (patch) => set((state) => ({
    user: state.user ? { ...state.user, ...patch } : null,
  })),
}))
```

**规则**：
- 全局 store 不超过 3 个（user / ui / 业务）
- store 内禁止异步操作，异步放 `services/` 层
- 组件内用 `useUserStore((s) => s.user)` 精确订阅，禁止 `useUserStore()` 全量订阅

#### 6.1 请求函数组织

```typescript
import { supabase } from '@/lib/supabase'

export async function fetchPostById(id: string): Promise<Post> { ... }

export async function createPost(data: CreatePostParams): Promise<Post> { ... }

export async function toggleLike(postId: string): Promise<{ liked: boolean; count: number }> { ... }
```

#### 6.2 错误处理规范

```typescript
// lib/api-error.ts
export class ApiError extends Error {
  constructor(
    public code: number,
    message: string,
    public userMessage?: string,
  ) {
    super(message)
  }
}

// services/post.ts
export async function fetchPosts(...) {
  const { data, error } = await supabase
    .from('posts')
    .select('*, profiles(nickname, avatar_url)')
    .range(offset, offset + pageSize - 1)

  if (error) {
    throw new ApiError(
      error.code,
      error.message,
      '获取帖子失败，请稍后重试',  // 用户可见文案
    )
  }

  return { list: data ?? [], total, hasMore, page }
}
```

#### 6.3 调用方统一处理

```typescript
// hooks/usePosts.ts
import useSWR from 'swr'
import { fetchPosts } from '@/services/post'

export function usePosts(page: number, tag?: string) {
  return useSWR(
    ['posts', page, tag],
    () => fetchPosts({ page, tag }),
    {
      revalidateOnFocus: false,
      errorRetryCount: 2,
      errorRetryInterval: 3000,
    }
  )
}
```

#### 6.4 禁止项

- ❌ 在组件内直接调用 `supabase.from()`（必须通过 `services/`）
- ❌ `try/catch` 吞掉错误不处理
- ❌ 在 `services/` 层操作 UI（如弹 Toast）
- ❌ 硬编码 API URL（使用环境变量）

### 七、Git 工作流规范

#### 7.1 分支策略

| 分支 | 用途 | 命名 |
|------|------|------|
| `main` | 生产代码，受保护 | — |
| `dev` | 开发集成分支 | — |
| `feature/*` | 功能开发 | `feature/post-detail` |
| `fix/*` | Bug 修复 | `fix/image-403` |
| `hotfix/*` | 线上紧急修复 | `hotfix/login-crash` |

#### 7.2 Commit Message 规范

采用 [Conventional Commits](https://www.conventionalcommits.org/)：

```
<type>(<scope>): <subject>

[body]

[footer]
```

| type | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat(post): 添加帖子详情页图片轮播` |
| `fix` | Bug 修复 | `fix(upload): 修复图片上传 403 错误` |
| `refactor` | 重构 | `refactor(api): 抽取统一请求层` |
| `style` | 样式调整 | `style(home): 瀑布流卡片间距优化` |
| `perf` | 性能优化 | `perf(image): 添加图片懒加载` |
| `docs` | 文档 | `docs: 更新技术方案文档` |
| `chore` | 构建/工具 | `chore: 升级 Next.js 到 14.2` |
| `test` | 测试 | `test(post): 添加帖子卡片单元测试` |

**规则**：
- subject 不超过 50 字
- 使用中文描述（团队统一）
- 禁止 `update`、`fix bug`、`修改` 等无意义 message
- 每次 commit 只做一件事

#### 7.3 PR 规范

- 标题：`[类型] 简要描述`
- 描述模板：
  ```markdown
  ## 改动内容
  - 改了什么
  - 为什么改

  ## 影响范围
  - 影响了哪些页面/组件

  ## 测试方式
  - 如何验证改动正确

  ## 截图（UI 变更必填）
  | 改动前 | 改动后 |
  |--------|--------|
  | ...    | ...    |
  ```
- PR 不超过 **500 行 diff**，超过必须拆分
- 至少 1 人 Code Review 通过

### 九、安全规范

#### 9.1 XSS 防护

| 场景 | 防护方式 |
|------|----------|
| 用户输入展示 | React 默认转义 JSX，禁止 `dangerouslySetInnerHTML` |
| URL 参数 | 使用 `encodeURIComponent` 编码 |
| HTML 富文本 | 使用 DOMPurify 过滤 |
| Markdown 渲染 | 使用安全的 Markdown 库 + sanitize |

#### 9.2 认证安全

| 规则 | 说明 |
|------|------|
| Token 存储 | Supabase SDK 自动管理，禁止手动存 cookie |
| 敏感信息 | 环境变量 `.env.local`，禁止硬编码 |
| API Key | Supabase anon key 可暴露，service key 禁止前端使用 |
| 文件上传 | 服务端校验 MIME type + 文件大小（≤ 10MB） |
| SQL 注入 | Supabase SDK 自动参数化查询，禁止拼接 SQL |

#### 9.3 输入校验

```typescript
// lib/validators.ts
export const validators = {
  title: (v: string) => v.trim().length >= 2 && v.trim().length <= 50,
  content: (v: string) => v.trim().length >= 10 && v.trim().length <= 2000,
  nickname: (v: string) => /^[\u4e00-\u9fa5a-zA-Z0-9_]{2,20}$/.test(v),
  price: (v: string) => /^¥?\d+(\.\d{1,2})?$/.test(v) || v === '',
}
```

**规则**：前端校验 = 用户体验，后端校验 = 安全底线，两者缺一不可。

### 十、性能规则

#### 10.1 Bundle 预算

| 指标 | 预算 | 检查方式 |
|------|------|----------|
| 首屏 JS | < 150KB (gzip) | `next build` 输出 + Bundle Analyzer |
| 单路由 JS | < 50KB (gzip) | Bundle Analyzer |
| CSS 总量 | < 30KB (gzip) | Build 输出 |
| 图片资源 | 单张 < 500KB | 上传时压缩 |
| 字体文件 | < 100KB | 使用系统字体或 subset |

#### 10.2 强制优化项

| 项目 | 实现 | 检查时机 |
|------|------|----------|
| 图片懒加载 | `loading="lazy"` | Code Review |
| 路由代码分割 | Next.js 自动 + `dynamic()` | Code Review |
| 组件按需引入 | Framer Motion `lazy` | Code Review |
| 列表虚拟化 | 超过 50 条使用虚拟列表 | 性能测试 |
| 防抖/节流 | 搜索 300ms、滚动 100ms | Code Review |
| Memo 优化 | `useMemo` / `useCallback` 仅用于复杂计算 | 性能瓶颈时 |

#### 10.3 禁止项

- ❌ 引入整个 lodash（用 `lodash-es/xxx` 按需引入）
- ❌ 引入整个 moment.js（用 dayjs 或 date-fns）
- ❌ 在渲染函数内创建对象/函数（每次渲染都会重建引用）
- ❌ 同步大循环（> 1000 次用 Web Worker）
- ❌ `useEffect` 依赖项遗漏（ESLint 规则强制检查）

```bash
# ✅ NEXT_PUBLIC_ 前缀 = 前端可访问
NEXT_PUBLIC_SUPABASE_URL=https://xxx.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJhbG...

# ✅ 无前缀 = 仅服务端可访问
SUPABASE_SERVICE_KEY=eyJhbG...
EDGEONE_DEPLOY_TOKEN=xxx
```

**规则**：
- `NEXT_PUBLIC_` 变量禁止存放敏感信息
- 所有环境变量必须在 `.env.example` 中声明
- 代码中使用 `process.env.XXX` 时必须提供默认值

### 十二、Code Review 检查清单

每次 PR Review 必须逐项检查，未通过项必须修复：

#### 功能正确性
- [ ] 功能符合需求描述
- [ ] 边界情况已处理（空数据、网络异常、超长输入）
- [ ] 错误提示对用户友好

#### 代码质量
- [ ] 命名规范（文件、变量、函数、类型）
- [ ] 单文件不超过 300 行
- [ ] 无 `any` 类型
- [ ] 无 `@ts-ignore`
- [ ] 无 `console.log`（生产代码）
- [ ] 无硬编码文案（中文直接写在代码中）
- [ ] 无重复代码（DRY 原则）

#### 性能
- [ ] 图片有 `loading="lazy"`
- [ ] 列表有唯一 `key`（禁止用 index 作 key，除非列表不变）
- [ ] 无不必要的 re-render（检查 useEffect 依赖）
- [ ] 大组件有 memo 优化（仅在必要时）

#### 安全
- [ ] 用户输入已校验
- [ ] 无 `dangerouslySetInnerHTML`
- [ ] 无敏感信息硬编码
- [ ] API Key 在环境变量中

#### 响应式
- [ ] 手机端（< 768px）正常显示
- [ ] PC 端（≥ 1024px）布局合理
- [ ] 触控目标 ≥ 44×44px

#### 无障碍
- [ ] 图标按钮有 `aria-label`
- [ ] 图片有 `alt` 文字
- [ ] 表单有 `<label>` 关联
