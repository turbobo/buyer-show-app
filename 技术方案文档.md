## 买家说 — 技术方案文档

### 一、项目概述

「买家说」是一个购物种草社区 H5 应用，用户可以发布购物体验、浏览好物推荐、互动评论点赞。面向移动端优先设计，部署到 EdgeOne Pages。

### 二、技术栈

| 层面 | 选型 | 版本 | 说明 |
|------|------|------|------|
| 框架 | Next.js (App Router) | 14.x | React 元框架，支持 SSG/SSR |
| 语言 | TypeScript | 5.4 | 类型安全 |
| 样式 | Tailwind CSS | 3.4 | 原子化 CSS |
| 动画 | Framer Motion | 11.x | 页面转场、微交互 |
| 状态 | Zustand | 4.5 | 轻量状态管理 (<1KB) |
| 后端 | Supabase | - | PostgreSQL + Auth + Storage（待接入） |
| 部署 | EdgeOne Pages | - | 静态导出，腾讯 CDN |

### 六、数据库设计（Supabase）

```sql
-- 用户资料
CREATE TABLE profiles (
  id UUID PRIMARY KEY REFERENCES auth.users(id),
  nickname TEXT NOT NULL,
  avatar_url TEXT DEFAULT '',
  bio TEXT DEFAULT '',
  post_count INT DEFAULT 0,
  follower_count INT DEFAULT 0,
  following_count INT DEFAULT 0,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 帖子
CREATE TABLE posts (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES profiles(id),
  title TEXT NOT NULL,
  content TEXT NOT NULL,
  images TEXT[] DEFAULT '{}',
  tags TEXT[] DEFAULT '{}',
  product_name TEXT DEFAULT '',
  price TEXT DEFAULT '',
  rating INT DEFAULT 5,
  like_count INT DEFAULT 0,
  comment_count INT DEFAULT 0,
  favorite_count INT DEFAULT 0,
  status TEXT DEFAULT 'active',
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 评论
CREATE TABLE comments (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  post_id UUID REFERENCES posts(id) ON DELETE CASCADE,
  user_id UUID REFERENCES profiles(id),
  content TEXT NOT NULL,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 点赞
CREATE TABLE likes (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  post_id UUID REFERENCES posts(id) ON DELETE CASCADE,
  user_id UUID REFERENCES profiles(id),
  created_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(post_id, user_id)
);

-- 收藏
CREATE TABLE favorites (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  post_id UUID NOT NULL REFERENCES posts(id) ON DELETE CASCADE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE(user_id, post_id)
);
```

#### 7.1 动态路由与静态导出

`next.config.mjs` 中 `output: 'export'` **仅在生产构建启用**：

```js
const isProd = process.env.NODE_ENV === 'production'
const nextConfig = {
  ...(isProd ? { output: 'export' } : {}),
  images: { unoptimized: true },
}
```

**原因**：开启 `output: 'export'` 后，dev server 也会按"全静态导出"行为校验 dynamic route。`/post/[id]/page.tsx` 通过 `generateStaticParams()` 仅在启动时拉取一次数据库帖子列表，dev 期间新发布的帖子（不在该列表里）访问详情页会报：

```
Error: Page "/post/[id]/page" is missing param "/post/<uuid>"
in "generateStaticParams()", which is required with "output: export" config.
```

dev 模式去掉 `export` 后，dynamic route 走运行时渲染，新帖立即可访问。

**生产环境已知限制**：`out/` 目录是 build 时的快照，部署后用户新发布的帖子在下次构建之前，详情页（直接访问 URL 或刷新）会 404。可选缓解方案：

1. 发布行为触发 EdgeOne Pages 重新构建（占用 500/月免费额度，新帖 1-2 min 可见）
2. 详情页改用 query 形式（`/post?id=xxx`）+ 客户端拉数据（牺牲 OG/SEO 预渲染）
3. 切换到 SSR 部署（`output` 不设 `export`，EdgeOne Pages 走函数模式，吃函数请求额度）

### 7.2 帖子编辑 / 删除归属校验

`updatePost(postId, params)` 与 `deletePost(postId)` 的契约：

1. 先 `auth.getUser()` 确认登录态，未登录直接抛 `请先登录`。
2. SELECT 原帖 `user_id`，与当前用户对比，不一致抛 `无权修改/删除此帖子`。
3. UPDATE/DELETE 时同时附带 `.eq('user_id', user.id)`，让 SQL/RLS 再兜一层底（防止 1→2 之间被绕过）。
4. UPDATE 时只写入显式传入的字段（`Partial<CreatePostParams>` 中 `!== undefined`），避免把可选字段误清空。

**为什么不只靠 RLS**：RLS 失败时 PostgREST 返回的是 `0 rows affected`（甚至直接 200 OK + 空数据），UI 层无法区分"无权"与"已删除"；service 层显式 SELECT 校验能给出准确的错误码与提示文案，对前端更友好。

**路由层**：`/post/[id]/edit` 复用 `[id]` 的 `generateStaticParams`，client 组件挂载时也做一次客户端归属校验（提前阻断 + 给出"无权修改"空状态页），不依赖服务端校验决定是否渲染表单。我的发布页（`/profile/posts`）通过 PostCard 右上角三点菜单入口跳转编辑页和触发删除二次确认弹窗。

**PostCard 作者菜单模式**：`PostCard` 新增可选 `ownerActions?: { onEdit, onDelete }` 参数；传入后卡片封面图右上角出现三点按钮（参考 Instagram / 小红书主流范式），点击展开 `编辑 / 删除` 下拉菜单，删除项用 coral 色突出危险感。三层点击隔离避免触发外层 `<Link>` 跳转：① 菜单容器 `onClick(e) => e.stopPropagation()`；② 三点按钮 `onClick` 自身 `stopPropagation + preventDefault`；③ 菜单项 `invoke` 再次 `stopPropagation + preventDefault` 后才调用回调。下拉菜单默认 150ms `scale + translateY` 过渡，外部点击（`mousedown` 全局监听 + `ref.contains` 判断）自动收起。封面图"N图"徽章在 ownerActions 存在时自动迁到左上角以避免遮挡。

### 7.3 账号管理契约

**昵称唯一性**：

- DB 兜底：`migration-v3-account.sql` 创建 `LOWER(nickname)` 函数式唯一索引（大小写不敏感，空字符串豁免）。历史重名通过 `ROW_NUMBER` 窗口函数自动追加 id 前 6 位重命名。
- service 层：`isNicknameAvailable(nickname, excludeUserId?)` 走 head + count；`updateProfile` 在 UPDATE 前先调 `isNicknameAvailable`，再做长度（2-20）/ bio（≤200）校验。
- 前端：编辑页对昵称输入做 350ms 防抖 + 状态机（`idle/checking/ok/taken/invalid`），实时给出可用性反馈，避免提交才报错。

**头像上传**：

- Storage 桶 `avatars` 的策略受权限问题阻塞，当前 `avatars` 走 base64 兜底（≤500KB），写到 `profiles.avatar_url`。
- Storage 解锁后切换路径：`avatars/{userId}/{timestamp}.webp`；UI 层 `<img>` 不变，只换 service 实现。

**注销账号（软注销）**：

`services/post.ts#deleteAccount` 三步：

1. `posts.update({ status: 'deleted' }).eq('user_id', uid)` —— 隐藏全部内容，前台查询统一过滤 `status='active'`。
2. `profiles.update({ nickname: '已注销用户_<6位>', avatar_url: '', bio: '' })` —— 不删行，避免 ON DELETE CASCADE 把外键关联（评论、点赞、收藏）一并抹掉触发计数器走偏。
3. `auth.signOut()` —— 客户端会话失效。

**真删除（auth.users 行）路径**：客户端无 service_role 权限，必须放 Edge Function。CASCADE 会自动清理 `profiles` 及所有外键依赖；但要额外清空对应帖子触发的计数（如被注销用户点过赞的帖子，likes 行被 CASCADE 删除时由 trigger 反扣 like_count，无需特殊处理）。

### 7.4 收藏模块（帖子 / 评论 / 标签）

`services/favorite.ts` 统一管理三类收藏，按领域拆开（不塞到 `post.ts`）：

| 函数 | 领域 | 行为 |
|------|------|------|
| `fetchUserFavorites` | 帖子 | JOIN `posts` 拉用户帖子列表 |
| `toggleCommentFavorite` | 评论 | 切换收藏，返回最新 `favorite_count`（DB 触发器维护） |
| `checkCommentFavorited` | 评论 | 检查当前用户是否已收藏 |
| `fetchUserFavoriteComments` | 评论 | 拉收藏评论列表（带 JOIN 的 user） |
| `toggleTagFavorite` | 标签 | 切换标签收藏，返回 `{ favorited: bool }` |
| `fetchUserFavoriteTags` | 标签 | 返回当前用户收藏的 tag 字符串数组（按收藏时间倒序） |
| `fetchUserFavoriteTagSet` | 标签 | 返回 `Set<string>`，给搜索页/标签云做高亮判断 |
| `fetchUserCommentsGroupedByPost` | 评论 | 拉当前用户评论按帖子聚合（Tab = 按帖子） |

**DB 触发器契约**：

- `favorite_comments` 行插入/删除时，`comments.favorite_count` 通过 `increment_comment_favorite_count` / `decrement_comment_favorite_count` 自动维护（迁移 `migration-v4-favorites-extra.sql`）。
- `favorite_tags` 不需要计数触发器（只是标签维度的用户偏好集合）。

**Supabase PostgREST 推断注意点**：

`favorites.select('created_at, post:posts(...)', ...)` 返回的行会被 PostgREST 推断为 `post: any[]`（数组而非单对象），需要在 `.map()` 时显式断言 `row as unknown as { post?: Post }`。这是 Supabase 生成类型的已知限制，不影响运行时语义（外键关联为多对一，DB 实际返回单行对象）。

**UI 层**：

- 搜索页：每个 HOT_TAG 拆成"主按钮（点击搜索）+ 右侧⭐按钮（切换收藏）"，`stopPropagation` 隔离点击，未登录走 `useAuthGuard` 弹 LoginSheet。
- 帖子详情页评论：`CommentItem` 右侧新增⭐按钮，`initialFavorited` 由外层列表一次性注入。
- 个人中心 `/profile/favorites`：3 Tab（帖子 / 评论 / 标签），Tab 切换时按需拉取对应数据。
- 个人中心 `/profile/comments`：2 Tab（按时间 / 按帖子），按帖子 Tab 用 `fetchUserCommentsGroupedByPost`，每组展示帖子封面+标题+该帖下所有评论。

### 7.5 Service 层领域拆分

`services/` 按**数据库表 / 业务领域**拆成 5 个文件，每个文件控制在 300 行内（AGENTS.md 强制阈值）：

| 文件 | 领域 | 职责 |
|------|------|------|
| `post.ts` | 帖子 | `fetchPosts` / `fetchPostDetail` / `createPost` / `updatePost` / `deletePost` / `toggleLike` / `toggleFavorite` / `checkPostFavorited` / `uploadImage` / `searchPosts` |
| `comment.ts` | 评论 | `fetchPostComments` / `addComment` / `fetchUserComments` / `fetchUserCommentsGroupedByPost` |
| `user.ts` | 用户 | `fetchCurrentUser` / `isNicknameAvailable` / `updateProfile` / `fetchUserPosts` / `toggleFollow` / `deleteAccount` |
| `favorite.ts` | 收藏（扩展域） | 评论/标签收藏：`toggleCommentFavorite` / `checkCommentFavorited` / `fetchUserFavoriteComments` / `toggleTagFavorite` / `fetchUserFavoriteTags` / `fetchUserFavoriteTagSet` / `fetchUserFavorites` |
| `auth.ts` | 认证 | 匿名/邮箱/手机号登录、登出、会话恢复、`onAuthStateChange` |

**依赖规则**：

- `auth.ts` 只依赖 `user.ts`（`fetchCurrentUser`），不依赖 `post.ts`，避免登录链路被无关业务模块拖累。
- `post.ts` 内的 `fetchPostDetail` 仍**内联**评论查询（避免反向依赖 `comment.ts`，造成循环）；`comment.ts` 的 `fetchPostComments` 供详情页之外的场景复用（如独立评论面板、管理后台）。
- `favorite.ts` 只处理**扩展域**收藏（评论/标签）。帖子收藏仍留 `post.ts`（与 `toggleLike` 同语义，且详情页组件一次 import 拿全）。

**文件阈值**：任何 service 文件增长接近 300 行时，按领域再切一刀。优先拆**表边界**（如将来若有 `notification.ts` / `report.ts` / `admin.ts`），避免切出"工具型 service"。

**调用方 import 风格**：按领域分组，不混写。示例（`/profile/favorites` 页）：

```ts
import { fetchUserFavorites, fetchUserFavoriteComments, fetchUserFavoriteTags, toggleTagFavorite } from '@/services/favorite'
```

不要在组件内 `import * as postApi from '@/services/post'`——会破坏 tree-shaking 并掩盖真实的依赖图。

### 7.6 管理员模块（MVP）

**DB 设计（`migration-v5-admin.sql`）**：

- `profiles` 新增 `role TEXT DEFAULT 'user' CHECK (role IN ('user','admin'))` 与 `status TEXT DEFAULT 'active' CHECK (status IN ('active','banned','deleted'))`。
- 两个 STABLE `SECURITY DEFINER` 辅助函数：`is_admin(uid)`、`has_any_admin()`。
- 三个 RPC 函数：
  - `promote_to_admin(target_uid)`：调用者=自己时仅在 `has_any_admin()=false` 允许（首次自举）；否则调用者必须是 admin。
  - `ban_user(target_uid)`：调用者必须是 admin；不能封自己；连带把目标所有 `status='active'` 的帖子置 `deleted`。
  - `unban_user(target_uid)`：调用者必须是 admin；不能解封自己。
- 新增 RLS `profiles_admin_all`：`USING (is_admin(auth.uid()))`，让管理员读写任意 profile。
- 索引：`idx_profiles_role` / `idx_profiles_status`。

**Service 层（`services/admin.ts`）**：

| 函数 | 行为 |
|------|------|
| `adminFetchUsers(page, search?)` | 直接 `.from('profiles').select('*')`（受 `profiles_select_all` 公开读策略保护）；搜索支持 `nickname.ilike.%kw%` 或 `id.eq.xxx` |
| `adminPromote(uid)` | `supabase.rpc('promote_to_admin')` |
| `adminBan(uid)` | `supabase.rpc('ban_user')` |
| `adminUnban(uid)` | `supabase.rpc('unban_user')` |

### 7.7 状态字段 SMALLINT 化（v6 / v7 迁移）

按「简单语义状态字段统一用整型 + 代码常量」原则（详见 `技术开发规范.md` §11），把 DB 中所有 `TEXT + CHECK IN (...)` 状态/角色字段迁为 `SMALLINT`。语义只在 `src/lib/constants.ts` 维护，DB 不再绑定字符串字面量：

| 表 / 字段 | 常量 | 取值 | 迁移脚本 |
|-----------|------|------|----------|
| `posts.status` | `POST_STATUS` | 0=正常 / 1=隐藏 / 2=删除 | `migration-v6-post-status-int.sql` |
| `profiles.role` | `USER_ROLE` | 0=用户 / 1=管理员 | `migration-v7-profile-tag-status-int.sql` |
| `profiles.status` | `USER_STATUS` | 0=正常 / 1=封禁 / 2=注销 | 同上 |
| `tags.status` | `TAG_STATUS` | 0=启用 / 1=已归档 / 2=已删除 | 同上 |

**迁移五步法**（每个字段都按这个顺序执行，避免 PostgreSQL `2BP01: cannot drop column ... because other objects depend on it`）：

1. `ADD COLUMN xxx_code SMALLINT NOT NULL DEFAULT 0`
2. `UPDATE ... SET xxx_code = 0/1/2 WHERE xxx = 'a'/'b'/'c'`
3. **解除依赖**：`DROP POLICY` / `DROP INDEX` / `DROP CONSTRAINT`（按 `\d+ table` 显示的依赖列表全部删干净）
4. `DROP COLUMN xxx` → `RENAME COLUMN xxx_code TO xxx`
5. **重建依赖**：索引、RLS、`SECURITY DEFINER` 函数全部重写一遍（旧函数体里的字符串字面量必须同步换成整数）

**v7 重建的对象**：

- 函数：`is_admin` / `has_any_admin` / `promote_to_admin` / `ban_user` / `unban_user` / `admin_create_tag` / `admin_rename_tag` / `admin_merge_tag` / `admin_delete_tag` / `sync_tags_dict` / `admin_init`
- RLS：`profiles_admin_all` / `tags_admin_all`
- 索引：`idx_profiles_role` / `idx_profiles_status` / `idx_tags_status`

**前端镜像改造**：

- `types/User.role` / `User.status` 由字符串联合类型改为 `number`
- `services/admin.ts`、`hooks/useAdminGuard.ts`、`/admin/users`、`/admin/tags`、`/profile`、`/profile/follow`、`/user/[id]` 等所有 `=== 'admin'` / `=== 'banned'` / `=== 'active'` 全部替换为常量比较
- `services/admin.ts#adminFetchTags` 在统计 `usage_count` 时同步把 `posts.status` 比较换成 `POST_STATUS.ACTIVE`（v6 的残留 bug，v7 顺手修了）

**为什么不一次性批量 RPC + RLS 替换**：PostgREST 缓存函数签名（参数/返回类型），DROP 旧函数 → CREATE 新函数比 `CREATE OR REPLACE` 更稳，避免触发 schema 缓存失效后的间歇性 404。

**v7 实施踩过的坑（已修）**：

- `tags_admin_all` 策略 USING 子句里调了 `is_admin(uuid)`，所以 `DROP FUNCTION is_admin` 之前必须一并 `DROP POLICY tags_admin_all`，否则报 `2BP01: cannot drop function ... because policy ... depends on function`。
- 推论：**任何跨表 RLS 依赖的 SECURITY DEFINER 函数，迁移前要把所有引用方策略一次性收集好集中 DROP**，再 DROP 函数，最后统一重建。

### 7.8 解封恢复帖子 bug（v8 修复）

**问题**：v5-admin 引入的 `ban_user` 会把用户所有 `status=0 (ACTIVE)` 帖子软删为 `status=2 (DELETED)`，但 `unban_user` 仅恢复 `profiles.status=ACTIVE`，未把帖子从 `DELETED` 还原成 `ACTIVE`。结果：用户解封后所有历史帖子仍不可见，前端列表/个人页空荡荡。

**根因**：v5 设计时把「ban」当作单向操作——只考虑了封禁时的连带处理，没保留还原所需信息（帖子原状态），也没在 unban 对称处理。

**修复（`supabase/migration-v8-fix-unban-restore-posts.sql`）**：

1. **新增 `posts.original_status SMALLINT`**（默认 NULL，仅在 ban 时填值，unban 后清空），用来暂存帖子被 ban 前的真实状态，解决「原帖子是 ACTIVE 还是管理员主动 HIDDEN」的二义性。
2. **重写 `ban_user`**：把所有非 DELETED 帖子的当前 `status` 暂存到 `original_status`，再统一置 `status=2`。已 DELETED 的帖子不动（避免覆盖 `original_status`）。
3. **重写 `unban_user`**：
   - 有 `original_status` 的：精确还原（原本 ACTIVE 还 ACTIVE，原本 HIDDEN 还 HIDDEN），并清空 `original_status`。
   - 历史补丁：旧 `ban_user` 留下的 `status=2` 且 `original_status IS NULL` 的帖子，一律还原为 ACTIVE（因为旧 ban_user 只处理 `status=0`，所以 `status=2` 一定源自 ACTIVE）。
4. **历史数据补丁**：把已被 ban 用户（`profiles.status=1`）的 `status=2` 帖子 `original_status` 设为 0，让老数据也能走「精确还原」路径。

**对称设计**：ban 与 unban 成对操作，`original_status` 暂存列仅服务于这一对，平时为 NULL 不污染业务查询（所有 `posts.status` 过滤 SQL 无需改）。

**前端无需改动**：`services/admin.ts#adminUnban` 仍走 RPC，前端乐观更新 `status: USER_STATUS.ACTIVE` 已正确。

### 7.9 索引重建（v9：修复首页/个人页慢查询）

**问题**：v6 迁移 `posts.status` TEXT → SMALLINT 时 DROP 了 4 个旧索引（`idx_posts_created_at` / `idx_posts_user_id` / `idx_posts_tags` / `idx_posts_status_created`），但**忘了按新 SMALLINT 列重建**。结果：

- 首页 `SELECT * FROM posts WHERE status=0 ORDER BY created_at DESC LIMIT 20` 走全表扫描
- 个人页 `WHERE user_id=? AND status=0` 走全表扫描
- 标签筛选 `WHERE tags @> ARRAY['xxx'] AND status=0` 走 GIN 全扫
- 详情页 `generateStaticParams`：`SELECT id FROM posts WHERE status=0 LIMIT 100` 全扫
- 评论列表 `WHERE post_id=? ORDER BY created_at ASC` 走 seq scan
- Supabase 免费实例磁盘 I/O 慢，自然卡（首页单次 1.5s+）

**修复（`supabase/migration-v9-rebuild-posts-indexes.sql`）**：按实际查询模式补建 **partial index**（仅覆盖 `status=0/1/2` 中常用值），既节省索引体积又让 Postgres 优先选 partial index 走 Index Only Scan。

| 索引 | 覆盖查询 | 体积 |
|------|---------|------|
| `idx_posts_status_created` ON `(created_at DESC) WHERE status=0` | 首页 feed / 搜索排序 | 仅 active 帖子 |
| `idx_posts_user_id_active` ON `(user_id, created_at DESC) WHERE status=0` | 个人页帖子列表 | 仅 active 帖子 |
| `idx_posts_tags_active` ON `USING GIN (tags) WHERE status=0` | 标签筛选 | 仅 active 帖子 |
| `idx_posts_id_active` ON `(id) WHERE status=0` | `generateStaticParams` Index Only Scan | 仅 active 帖子 id |
| `idx_comments_post_id_created` ON `(post_id, created_at ASC)` | 评论列表 | 全量 |
| `idx_comments_user_id_created` ON `(user_id, created_at DESC)` | 用户最近评论 | 全量 |
| `idx_likes_post_user` / `idx_favorites_post_user` / `idx_favorite_tags_user_tag` / `idx_favorite_comments_user_comment` | 点赞/收藏唯一性校验 | 全量 |
| `idx_follows_follower` / `idx_follows_following` | 关注双向计数 | 全量 |
| `idx_search_history_user_created` | 搜索历史 | 全量 |

**为什么用 partial 而不是全量**：业务查询 95%+ 都是 `WHERE status=0`（active），HIDDEN/DELETED 帖子几乎不查。partial index 只覆盖 status=0，体积最小、cache 命中率最高。

**执行要点**：

- 用 `CREATE INDEX CONCURRENTLY` 避免锁表（Supabase SQL Editor 单语句执行天然不在事务内，可正常跑）
- 跑完 `ANALYZE` 让查询计划器立即看到新索引统计
- 索引加完后**立即生效**，无需重启 PostgREST 或前端

**预期效果**：首页列表从 1.5s+ 降到 50-150ms（Supabase 免费实例实测），个人页、标签筛选、评论列表同等量级改善。

**验证（Supabase SQL Editor）**：

```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM posts WHERE status = 0 ORDER BY created_at DESC LIMIT 20;
-- 期望：Index Scan using idx_posts_status_created

EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM posts WHERE user_id = '<some_uid>' AND status = 0 ORDER BY created_at DESC;
-- 期望：Index Scan using idx_posts_user_id_active
```

### 八、搜索匹配规则

`services/post.ts#searchPosts` 当前匹配策略:

| 字段 | 操作符 | 匹配方式 |
|------|--------|----------|
| `title` | `ilike` | 大小写不敏感模糊包含（`%kw%`） |
| `content` | `ilike` | 大小写不敏感模糊包含 |
| `product_name` | `ilike` | 大小写不敏感模糊包含 |
| `tags` (TEXT[]) | `cs` (contains) | 数组**整段**包含，如 `{数码}` |

PostgREST 通过 `.or(...)` 组合多字段，多条件用逗号分隔。关键词中的 `,`、`(`、`)` 会破坏 or 表达式，调用前会先剥除这三个字符。

**为什么 tags 不用 ilike**：`tags` 是 `TEXT[]`，无法直接 ilike；用 `cs` 整段匹配能命中 `["数码","耳机"]` 这类数组里的标签。代价是只能完全匹配单个 tag，不能模糊（如「数」搜不到「数码」），符合产品「点热门标签 = 精确分类筛选」的预期。

**已知限制 / 后续可优化**：

- 大数据量下 `ilike '%kw%'` 走全表扫描，需建 `pg_trgm GIN` 索引，或升级到 PostgreSQL 全文检索（`tsvector` + `to_tsquery`）。
- `tags` 列建议建 GIN 索引（首页标签筛选已建，搜索复用）。
- 关键词转义目前是黑名单（剥除 `,()`），无法防御 `%`、`_` 这类 ilike 通配符；后续若加入用户自定义复杂查询需做白名单转义。

### 九、错误处理契约

错误处理是工程质量的底线，详细规范见 `技术开发规范.md` §6.2。架构层登记如下：

**核心文件**：

| 文件 | 职责 |
|------|------|
| `src/lib/error-codes.ts` | 业务错误码统一表（通用 1xxx / 认证 2xxx / 校验 3xxx / 资源 4xxx / 业务 5xxx） |
| `src/lib/api-error.ts` | 统一异常类 `ApiError`，封装 code / userMessage / cause |

**抛错与捕获形式**：

- `services/` 层只抛 `ApiError` 实例，禁止抛 `string` 或裸 `Error`
- 调用方用 `ApiError.is(err)` 类型守卫识别业务错误，未识别走兜底文案 + `console.error('[scope]', err)`
- `services/` 层禁止操作 UI（弹 Toast、跳转），UI 关注点全部归调用方

**用户提示分级**：

| 错误类型 | 提示形式 |
|---------|---------|
| 表单实时校验 | 内联红字 + 按钮上方 hint |
| 业务可恢复 | Toast 自动消失 |
| 认证失败 | 弹 LoginSheet 或跳 `/login` |
| 阻断性错误 | 整页错误态 + 返回按钮 |
| 全局兜底 | `app/error.tsx` 错误边界 |

**前后端约束镜像**：数据库 CHECK 约束（如 `posts_content_check: char_length(content) >= 10`）必须在 `services/` 层用对应错误码（`CONTENT_TOO_SHORT`）做前置校验，避免裸 PostgREST 错误信息泄漏给用户。

### 十、认证与会话持久化

**目标**：刷新页面、关闭重开、跨 Tab 操作都不应让已登录用户回到未登录状态。

**Supabase 客户端配置**（`src/lib/supabase.ts`）：

| 选项 | 值 | 说明 |
|------|----|------|
| `persistSession` | `true` | 写入 localStorage，刷新可恢复 |
| `autoRefreshToken` | `true` | access_token（默认 1h 过期）后台续签 |
| `detectSessionInUrl` | `true` | OAuth/邮件回调时从 URL 解析 session |
| `storageKey` | `'buyer-show-auth'` | 自定义键名，避免与同域其他项目冲突 |
| `flowType` | `'pkce'` | 推荐的 OAuth 流程，安全性更高 |

**架构分层**：

```
RootLayout (app/layout.tsx)
  └── AuthProvider (components/AuthProvider.tsx)   ← 全局唯一挂载
        └── useAuthInit (hooks/useAuthInit.ts)
              ├── 启动：supabase.auth.getSession() → 同步 profile → store.setUser
              ├── 订阅：onAuthStateChange (SIGNED_IN/OUT/TOKEN_REFRESHED/USER_UPDATED)
              └── 完成后：store.setAuthReady(true)
```

关键约束：

- `useAuthInit` 必须挂在根布局，**不得**只挂在某个页面 —— 否则刷新到其他路由时不会触发会话恢复
- 业务页面只读 `useUserStore`（`user`、`isLoggedIn`、`authReady`），**不得**直接调用 `supabase.auth.getSession()`
- 任何有"未登录跳转 / 弹未登录 Sheet"逻辑的 effect，必须先判断 `if (!authReady) return`，避免刷新时 race
- 渲染层若关心"会话恢复中"状态，使用 `authReady` 渲染骨架占位，避免"已登录闪一下未登录"的视觉抖动

**Zustand 状态契约**（`src/store/user.ts`）：

| 字段 | 含义 |
|------|------|
| `user` | 当前 profile，未登录为 `null` |
| `isLoggedIn` | 派生值，`!!user` |
| `authReady` | 启动时会话恢复是否完成；初始 `false`，AuthProvider 完成 `getSession` 后置 `true` |
| `setUser(user)` | 由 `useAuthInit` 与各登录服务调用 |
| `setAuthReady(b)` | 仅 `useAuthInit` 内部调用 |
| `logout()` | 清空状态；实际 supabase signOut 由 `auth-helpers.quickLogout` 负责 |
